# 缺陷分析器 (defect-analyzer)

## SKILL技能触发机制

### 触发条件

当用户输入符合以下条件时，系统会自动触发defect-analyzer技能：

1. **问题描述关键词**：输入中包含"问题"、"bug"、"缺陷"、"崩溃"、"错误"、"异常"等关键词
2. **问题类型标识**：输入中包含"问题类型"、"类型"等标识
3. **复现概率标识**：输入中包含"复现概率"、"概率"、"出现频率"等标识
4. **复现手法标识**：输入中包含"复现手法"、"复现步骤"、"如何复现"等标识
5. **Product Flavor标识**：输入中包含"flavor"、"Product Flavor"等标识

### 调用方法

#### 直接调用
用户可以直接描述问题，系统会自动识别并触发技能：

```
问题描述：应用在点击登录按钮后崩溃
问题类型：崩溃
复现概率：100%
复现手法：1. 打开应用 2. 输入用户名密码 3. 点击登录按钮
Product Flavor：xxx
```

#### 显式调用
用户也可以通过明确指令调用技能：

```
使用defect-analyzer技能分析以下问题：
问题描述：应用在点击登录按钮后崩溃
问题类型：崩溃
复现概率：100%
复现手法：1. 打开应用 2. 输入用户名密码 3. 点击登录按钮
Product Flavor：xxx
```

### 技能协作

defect-analyzer技能可以与其他技能协作：

1. **与defect-document-generator协作**：当分析完成后，推荐用户使用defect-document-generator技能生成缺陷文档
2. **与SKILL技能协作**：当需要更复杂的技能调用时，通过SKILL技能进行协调

### 响应格式

技能触发后，系统会按照以下格式响应：

1. **分析结果**：提供对问题的分析结果
2. **相关文档**：列出匹配的相关文档
3. **解决方案方向**：提供建议的解决方案方向
4. **下一步建议**：推荐用户使用defect-document-generator技能生成缺陷文档

### 连续触发

支持连续触发机制，当用户对分析结果有进一步问题时，系统会继续使用defect-analyzer技能进行分析。

## 概述

该技能用于Android应用开发中的bug或者问题分析，通过用户提供的问题描述、问题类型、复现概率、复现手法等信息，检索当前技能下的相关md文档，快速定位问题并提供解决方案方向。

**重要说明**：defect-analyzer里的md文档内容仅供参考和借鉴，具体实现请根据实际情况进行调整，不要完全按照文档内容进行代码修改。

## 通用处理流程

### 设计目的

通用处理流程文档（assets目录下的文档）用于在references目录下没有相关历史问题文档时，为用户提供标准的Android应用问题分析和解决流程。这些文档基于Android开发最佳实践和常见问题场景编写，涵盖了ANR、崩溃、性能、UI、网络等主要问题类型。

### 使用场景

1. **首次遇到的问题**：当某个问题首次出现，还没有历史处理文档时
2. **通用问题分析**：当问题属于常见类型，但缺乏具体案例时
3. **流程指导**：当需要标准的问题分析和解决流程时

### 文档内容结构

每个通用处理流程文档包含以下内容：

1. **问题类型识别**：详细列出该类型问题的常见子类和特征
2. **分析工具**：推荐使用的Android开发工具和调试方法
3. **分析步骤**：系统化的问题分析流程和检查点
4. **解决方案**：针对不同子类问题的具体解决方案和代码示例
5. **预防措施**：避免问题再次发生的最佳实践和建议
6. **常见问题及解决**：典型案例分析和解决方法
7. **检查清单**：完整的检查点列表，确保不遗漏关键步骤

### 使用方式

当references目录下没有相关文档时，技能会自动：

1. 根据用户的问题类型，加载对应的通用处理流程文档
2. 读取文档内容，提取与用户问题相关的部分
3. 结合用户提供的问题描述、复现手法等信息，提供针对性的分析和建议
4. 明确告知用户当前使用的是通用处理流程

### 与references文档的区别

| 特性 | references文档 | assets通用流程文档 |
|------|---------------|------------------|
| 内容来源 | 历史实际问题的处理记录 | Android开发最佳实践和常见场景 |
| 针对性 | 针对特定问题，具有高度针对性 | 针对问题类型，具有通用性 |
| 更新频率 | 随着新问题的出现而增加 | 相对稳定，定期优化 |
| 使用优先级 | 优先使用 | references无相关文档时使用 |

### 持续优化

通用处理流程文档会根据以下情况持续优化：

1. 用户反馈：根据用户使用反馈改进文档内容
2. 新技术：随着Android系统和新技术的更新而更新
3. 最佳实践：根据Android开发最佳实践的变化而调整
4. 问题统计：根据常见问题类型的统计结果优化文档结构

## 工作流程

1. **接收缺陷信息**：获取用户提供的缺陷问题信息
2. **输入验证**：检查是否提供了所有必需的基本信息（问题描述、问题类型、复现概率、复现手法、Product Flavor），只要缺少其中任意一个信息，对话将停止执行并提示完整的输入格式。
3. **文档触发机制**：使用基于规则的文档触发机制，直接在SKILL.md中定义文档检索规则，避免使用脚本，提高性能。
4. **执行分析**：根据预定义的规则进行分析，包括：
   - Product Flavor 处理：根据用户提供的 Product Flavor，确定文档检索的 flavor 目录
   - 问题类型匹配：根据用户提供的问题类型，在 flavor 目录下定位到对应的文档分类
   - 相关文档检查：检查对应分类下是否存在相关文档
   - 关键词匹配：使用问题描述中的关键词在相关文档中进行匹配
   - 结果分析：提取匹配度最高的文档内容，分析解决方案方向
   - **通用处理流程回退**：如果references目录下没有相关文档，根据问题类型加载assets目录下的通用处理流程文档
5. **提供建议**：向用户提供分析结果和建议的解决步骤

## 文档触发机制

### 自动检索原理

使用AI自动检索references目录下的文档，避免使用固定规则列表，提高性能和可扩展性。工作原理如下：

1. **目录结构分析**：AI自动分析references目录下的文件结构，了解文档的组织方式
2. **文档内容索引**：AI自动索引所有文档的内容，建立关键词到文档的映射
3. **智能匹配**：当用户输入缺陷信息时，AI自动提取关键词并匹配相关文档
4. **动态更新**：当references目录下添加新文档时，AI自动更新索引，无需手动维护

### 性能优化

1. **智能索引**：使用AI建立智能索引，提高文档检索速度
2. **缓存机制**：缓存检索结果，提高重复查询的响应速度
3. **并行处理**：使用AI并行处理多个文档，提高检索效率
4. **增量更新**：当添加新文档时，只更新增量部分，避免全量索引

### 优势

1. **可扩展性**：文档数量增加时，无需手动更新规则，AI自动处理
2. **高效性**：AI索引和检索速度快，即使文档数量增加也能保持高性能
3. **准确性**：AI能够理解文档内容，提高匹配准确性
4. **灵活性**：能够处理各种类型的缺陷，不受固定规则的限制

### 实现方式

1. **目录扫描**：AI定期扫描references目录，发现新文档
2. **内容分析**：AI分析文档内容，提取关键词和主题
3. **索引构建**：AI构建文档索引，建立关键词到文档的映射
4. **查询处理**：当用户输入缺陷信息时，AI提取关键词并查询索引
5. **结果排序**：AI根据匹配度对结果进行排序，返回最相关的文档

### 强制处理要求

1. **文档规范**：所有新增文档必须遵循统一的文档格式，确保AI能够正确分析
2. **内容质量**：文档内容必须清晰、详细，确保AI能够准确理解
3. **目录结构**：保持references目录结构的一致性，确保AI能够正确扫描
4. **定期维护**：定期检查文档质量和索引状态，确保系统正常运行

## 缺陷分析规则

### 输入验证规则

1. **验证时机**：在接收缺陷信息后立即进行验证
2. **验证内容**：检查是否提供了所有必需的四个基本信息
3. **验证失败处理**：
   - 停止执行后续分析步骤
   - 生成清晰的错误提示，列出所有缺失的必需信息
   - 提示用户重新提供完整的信息
4. **验证通过条件**：四个必需信息全部提供且内容不为空

### 分析流程

1. **Product Flavor 匹配**：根据用户提供的 Product Flavor，定位到对应的 flavor 目录
2. **问题类型匹配**：根据用户提供的问题类型，使用问题类型映射表找到对应的文档分类文件夹
3. **相关文档检查**：检查对应分类下是否存在相关文档
   - 若没有相关文档，进入通用处理流程（见步骤4a）
   - 若有相关文档，继续执行步骤4b
4. **分支处理**：
   - **4a. 通用处理流程**（当references目录下没有相关文档时）：
     - 根据问题类型映射，加载assets目录下对应的通用处理流程文档
     - 读取通用处理流程文档内容
     - 根据用户问题信息，从通用流程文档中提取相关的分析和解决步骤
     - 整合分析结果，提供基于通用流程的解决方案方向
   - **4b. references文档处理**（当references目录下有相关文档时）：
     - 关键词提取：从问题描述和复现手法中提取关键术语
     - 文档匹配：在对应分类下的文档中，查找包含关键词的文档
     - 内容提取：提取匹配度最高的文档内容，分析解决方案方向
5. **结果生成**：整合分析结果，提供清晰的解决方案方向
6. **结果说明**：明确告知用户当前使用的是通用处理流程还是历史问题文档

### 关键词匹配规则

- 优先匹配文档标题中的关键词
- 其次匹配文档内容中的关键词
- 支持多关键词组合匹配
- 按匹配度排序返回结果
- 若没有找到匹配的文档，不执行references文件的内容提取和结果分析

### 无相关文档处理

1. **识别条件**：
   - 对应分类下没有任何文档
   - 有文档但没有与问题描述匹配的文档
2. **处理逻辑**：
   - 跳过references文件的执行
   - 根据问题类型加载assets目录下的通用处理流程文档
   - 使用通用处理流程文档进行分析
   - 向用户提供基于通用流程的分析结果
   - 鼓励用户使用defect-document-generator技能生成新的缺陷文档

## 日志使用规范

### Android原生Log组件

在缺陷修复过程中，如果需要添加或修改日志，使用Android原生Log类进行日志打印。

#### 日志级别说明

| 级别 | 常量 | 值 | 使用场景 |
|------|------|-----|---------|
| VERBOSE | Log.VERBOSE | 2 | 详细的调试信息，通常只在开发阶段使用 |
| DEBUG | Log.DEBUG | 3 | 调试信息，帮助开发者了解程序的运行状态 |
| INFO | Log.INFO | 4 | 一般信息，反映程序的正常运行情况 |
| WARN | Log.WARN | 5 | 潜在问题或非关键错误，提示开发者注意 |
| ERROR | Log.ERROR | 6 | 严重错误，影响程序的正常运行，需要及时处理 |

#### 日志使用示例

```java
// AI_AGENT_PATCH_START
// 使用Log打印日志
// AI_AGENT_PATCH_MODIFY
Log.v(TAG, "Verbose message");           // 详细日志
Log.d(TAG, "Debug message");              // 调试日志
Log.i(TAG, "Info message");               // 信息日志
Log.w(TAG, "Warning message");            // 警告日志
Log.e(TAG, "Error message: " + exception);   // 错误日志（带异常）
// AI_AGENT_PATCH_END
```

```kotlin
// AI_AGENT_PATCH_START
// 使用Log打印日志
// AI_AGENT_PATCH_MODIFY
Log.v(TAG, "Verbose message")
Log.d(TAG, "Debug message")
Log.i(TAG, "Info message")
Log.w(TAG, "Warning message")
Log.e(TAG, "Error message: " + exception);
// AI_AGENT_PATCH_END
```

#### 日志使用原则

1. **日志级别选择**：根据日志内容的重要性选择合适的日志级别
2. **敏感信息保护**：禁止在日志中打印敏感信息（密码、token等）
3. **TAG命名规范**：使用类名作为TAG，格式为：`private static final String TAG = "ClassName";`
4. **性能考虑**：避免在循环中频繁打印日志，建议使用BuildConfig.DEBUG控制调试日志输出

## 技能协作

该技能与 `skills/defect-document-generator` 技能协作，当用户处理完缺陷后，`skills/defect-document-generator` 技能会生成新的缺陷文档，并自动更新该技能的缺陷分类和文档库。

## 通用流程扩展指南

### 添加新的问题类型

当需要为新的问题类型添加通用处理流程时，按照以下步骤操作：

1. **创建新的通用流程文档**：
   - 在assets目录下创建新的md文档
   - 文件命名遵循：{问题类型}_analysis.md（小写字母，下划线分隔）
   - 例如：`memory_analysis.md`、`database_analysis.md`

2. **文档内容结构**：
   - 问题类型识别
