# Android 日志规范检查器

## 概述

此技能提供Android应用日志规范的全面检查功能，帮助开发者确保日志输出符合XBH项目的统一规范要求，包括日志格式验证、日志级别使用检查、日志安全性审查等。通过系统化的检查流程，确保日志信息完整、规范、安全。

## 检查触发条件

在以下场景触发此技能：
- 用户明确请求检查日志规范
- 用户提到"日志检查"、"log检查"、"日志规范"等关键词
- 用户询问日志格式是否正确
- 用户需要验证日志是否包含必要的六要素
- 用户关心日志安全性问题

## 日志格式规范

### 1. 标准日志格式

所有模块写入 Logcat 的日志采用统一格式：

```
[日期][日志级别][模块名称][进程ID:线程ID][文件类名:行数][日志内容]
```

**示例：**
```
[2025-08-11 22:19:43.549][E][XbhLiveTv][4314:2][null:0][getBootSourceIndex: specifiedSourceId=0]
[2025-08-11 22:19:43.736][W][Source][4294:68][XbhLogger.java:176][Method: PlatformSourceManagerImpl.getSrcDet;Args: arg0=35;Return: true;Cost: 10 ms]
```

### 2. 格式字段说明

| 字段 | 格式 | 说明 | 示例 |
|------|------|------|------|
| 日期 | [YYYY-MM-DD HH:mm:ss.SSS] | 精确到毫秒的时间戳 | [2025-08-11 22:19:43.549] |
| 日志级别 | [级别] | E(错误)、W(警告)、I(信息)、D(调试)、V(详细) | [E]、[W] |
| 模块名称 | [模块名] | 功能模块标识 | [XbhLiveTv]、[NFC]、[Source] |
| 进程ID:线程ID | [PID:TID] | 进程和线程标识 | [4314:2]、[4294:68] |
| 文件类名:行数 | [文件名:行号] | 日志输出的代码位置 | [XbhLogger.java:176]、[null:0] |
| 日志内容 | [内容] | 具体的日志信息 | [getBootSourceIndex: specifiedSourceId=0] |

### 3. 日志级别使用规范

根据日志类型选择合适的级别：

```java
public enum LogCategory {
    OPERATION(0),    // 操作日志 - 使用 I 级别
    RUNNING(1),      // 运行日志 - 使用 D 级别
    SECURITY(2),     // 安全日志 - 使用 W 级别
    DEBUG(3);        // 调试日志 - 使用 V 级别
    
    private final int code;
    
    LogCategory(int code) {
        this.code = code;
    }
    
    public int getCode() {
        return code;
    }
}
```

**级别选择原则：**
- **E (Error)**: 严重错误，影响功能正常运行
- **W (Warning)**: 警告信息，潜在问题或不影响主功能
- **I (Info)**: 重要操作信息，关键业务流程
- **D (Debug)**: 调试信息，开发阶段的详细信息
- **V (Verbose)**: 冗余信息，最详细的调试信息

## 检查流程

### 步骤 1：识别日志范围

首先确定需要检查的日志范围：

1. **确定检查对象**
   - 识别用户提供的日志文件或日志片段
   - 如果是代码文件，分析日志调用代码
   - 确定涉及的所有日志输出点

2. **收集上下文信息**
   - 读取相关的日志配置文件
   - 检查日志工具类的实现
   - 了解项目的日志架构

3. **识别日志类型**
   - 确定是Android原生日志还是自定义日志
   - 识别日志的调用链路
   - 确定日志的输出目标

### 步骤 2：日志格式检查

验证日志是否符合统一格式规范：

1. **时间戳格式检查**
   - 检查日期格式是否为 YYYY-MM-DD HH:mm:ss.SSS
   - 验证时间戳是否精确到毫秒
   - 确认时区是否正确

2. **日志级别检查**
   - 验证日志级别是否为有效值（E/W/I/D/V）
   - 检查级别使用是否符合场景
   - 确认是否有滥用高级别日志的情况

3. **模块名称检查**
   - 检查模块名称是否清晰明确
   - 验证模块名称是否统一
   - 确认是否便于日志过滤和定位

4. **进程线程信息检查**
   - 验证PID:TID格式是否正确
   - 检查进程线程信息是否完整
   - 确认便于问题追踪和定位

5. **代码位置检查**
   - 检查文件名:行号格式是否正确
   - 验证是否便于快速定位代码
   - 确认代码位置信息是否准确

6. **日志内容检查**
   - 检查日志内容是否清晰明了
   - 验证是否包含必要的上下文信息
   - 确认内容格式是否规范

**日志格式参考：** 加载 `references/log_format_standards.md` 获取详细的格式规范说明。

### 步骤 3：日志六要素检查

验证日志是否包含必要的安全审计六要素：

根据安全审计要求，日志应包含以下六要素：

1. **事件发生的时间 ✓**
   - 检查日志是否包含精确的时间戳
   - 验证时间格式是否符合规范
   - 确认时间信息完整且准确

2. **用户ID ✓**
   - 检查日志是否包含用户标识信息
   - 验证用户ID是否脱敏处理
   - 确认便于追踪用户操作

3. **访问发起端地址或标识 ✓**
   - 检查日志是否包含设备标识或网络地址
   - 验证终端信息的完整性
   - 确认包含端口、网络地址或通信设备等标识

4. **事件类型 ✓**
   - 检查日志是否明确标识事件类型
   - 验证事件类型分类是否合理
   - 确认便于日志分类和检索

5. **被访问的资源名称 ✓**
   - 检查日志是否包含操作的资源信息
   - 验证资源名称是否清晰
   - 确认便于资源访问追踪

6. **事件的结果 ✓**
   - 检查日志是否包含操作结果
   - 验证成功/失败状态是否明确
   - 确认便于问题诊断和审计

**六要素检查清单：**
- [ ] 时间：精确到毫秒的时间戳
- [ ] 用户ID：用户标识（已脱敏）
- [ ] 访问端标识：设备ID、IP地址或端口信息
- [ ] 事件类型：操作类型、业务类型
- [ ] 资源名称：被访问的资源或功能
- [ ] 事件结果：成功、失败、错误码

**六要素参考：** 加载 `references/log_six_elements.md` 获取详细的六要素说明和示例。

### 步骤 4：日志打印链路检查

验证日志打印的调用链路是否正确：

1. **Android原生链路**
   - 中间件与APK直接调用Android原生日志接口
   - 检查是否使用 Log.e()、Log.w()、Log.i()、Log.d()、Log.v()
   - 验证日志级别参数是否正确

2. **EYWA模块链路**
   - EYWA模块调用自定义日志接口 XLOGI
   - 验证调用链路：
     ```
     XLOGI(...)
       → AndroidPrintLog(...)
         → android_printLog(...)
           → android_log_print(...)
     ```
   - 检查自定义接口的实现是否正确

**调用链路参考：** 加载 `references/log_printing_workflow.md` 获取详细的调用链路说明。

### 步骤 5：日志安全性检查

检查日志是否存在安全隐患：

1. **敏感信息检查**
   - 检查日志中是否包含密码、密钥、Token等敏感信息
   - 验证用户隐私数据是否脱敏处理
   - 确认个人身份信息是否加密或脱敏

2. **生产环境安全**
   - 检查Debug级别日志是否在生产环境关闭
   - 验证敏感操作日志是否有权限控制
   - 确认日志文件是否有访问权限保护

3. **日志注入风险**
   - 检查日志内容是否进行了必要转义
   - 验证是否防止日志伪造攻击
   - 确认是否避免了日志注入漏洞

**安全性检查参考：** 加载 `references/log_security_checklist.md` 获取详细的安全检查要点。

### 步骤 6：日志性能检查

检查日志对性能的影响：

1. **日志量控制**
   - 检查是否有大量冗余日志
   - 验证日志级别设置是否合理
   - 确认是否避免了日志洪泛

2. **性能影响**
   - 检查日志是否在主线程打印
   - 验证日志格式化是否高效
   - 确认是否使用了异步日志机制

3. **存储优化**
   - 检查日志文件是否有大小限制
   - 验证日志文件是否有轮转机制
   - 确认过期日志是否有清理机制

### 步骤 7：生成检查报告

整理检查结果，生成清晰的检查报告：

1. **报告结构**
   - 格式规范性：日志格式是否符合标准
   - 六要素完整性：是否包含必要的审计信息
   - 安全合规性：是否存在安全隐患
   - 性能影响：日志对性能的影响评估
   - 改进建议：具体的优化和修复建议

2. **问题分级标准**
   - **严重**：包含敏感信息、缺失关键审计要素、安全漏洞
   - **警告**：格式不规范、级别使用不当、性能问题
   - **建议**：优化建议、最佳实践推荐

3. **修复建议**
   - 对于严重问题，提供具体的修复方案
   - 对于警告和建议，提供改进方向
   - 引用相关的参考文档和最佳实践

## 检查示例

### 示例 1：日志格式检查

**输入日志：**
```
[2025-08-11 22:19:43.549][E][XbhLiveTv][4314:2][null:0][getBootSourceIndex: specifiedSourceId=0]
```

**检查结果：**
- ✓ 时间戳格式正确：2025-08-11 22:19:43.549
- ✓ 日志级别正确：E (Error)
- ✓ 模块名称清晰：XbhLiveTv
- ✓ 进程线程信息完整：4314:2
- ⚠ 代码位置缺失：[null:0] 应该包含具体的文件名和行号
- ✓ 日志内容清晰：getBootSourceIndex: specifiedSourceId=0

**修复建议：**
确保日志输出时包含准确的代码位置信息，便于快速定位问题。

### 示例 2：六要素完整性检查

**输入日志：**
```
[2025-08-11 22:19:43.736][W][Source][4294:68][XbhLogger.java:176][Method: PlatformSourceManagerImpl.getSrcDet;Args: arg0=35;Return: true;Cost: 10 ms]
```

**六要素检查：**
- ✓ 时间：2025-08-11 22:19:43.736
- ⚠ 用户ID：日志中未明确包含用户标识
- ⚠ 访问端标识：日志中未包含设备或网络标识
- ✓ 事件类型：方法调用 getSrcDet
- ✓ 资源名称：PlatformSourceManagerImpl
- ✓ 事件结果：Return: true

**改进建议：**
在日志中增加用户ID和访问端标识信息，确保六要素完整，满足安全审计要求。

### 示例 3：安全性检查

**问题代码：**
```java
Log.d("Login", "User login: username=" + username + ", password=" + password);
```

**检查结果：**
- **严重**：日志中包含密码明文，存在严重安全隐患

**修复建议：**
```java
Log.d("Login", "User login: username=" + username + ", userId=" + userId);
```
或对敏感信息进行脱敏：
```java
Log.d("Login", "User login: username=" + maskSensitiveInfo(username));
```

## 注意事项

1. **检查优先级**
   - 优先检查严重的安全问题（敏感信息泄露）
   - 其次检查六要素完整性（审计合规）
   - 最后检查格式规范和性能优化

2. **上下文理解**
   - 充分理解业务场景和安全要求
   - 考虑不同环境的日志策略差异
   - 避免过度检查和误报

3. **实用建议**
   - 提供可操作的修复建议
   - 给出具体的代码示例
   - 考虑修复的成本和收益

4. **团队规范**
   - 遵循团队的日志规范
   - 考虑项目的特殊要求
   - 保持与现有日志风格一致

## 日志最佳实践

1. **日志级别使用**
   - Error：只记录真正的错误，不用于记录业务异常
   - Warning：记录潜在问题和需要注意的情况
   - Info：记录关键业务流程和重要操作
   - Debug：记录开发调试信息，生产环境应关闭
   - Verbose：记录详细调试信息，仅开发环境使用

2. **日志内容规范**
   - 包含足够的上下文信息
   - 使用结构化的日志格式
   - 避免日志信息过于冗长
   - 便于日志分析和问题定位

3. **性能优化**
   - 避免在循环中大量打印日志
   - 使用异步日志机制
   - 合理设置日志级别
   - 定期清理过期日志

4. **安全合规**
   - 敏感信息脱敏或加密
   - 生产环境关闭Debug日志
   - 日志文件权限控制
   - 定期审计日志内容
