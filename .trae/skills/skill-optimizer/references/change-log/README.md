# SKILL优化日志

## 概述

此目录用于存储skill-optimizer对所有SKILL进行优化的详细日志记录。

## 目录结构

```
change-log/
├── README.md
├── <skill-name-1>/
│   ├── <problem-description>-yyyy-mm-dd.md
│   └── ...
├── <skill-name-2>/
│   └── ...
└── ...
```

## 命名规范

- **SKILL目录名**: 使用小写字母和连字符,与SKILL名称一致
  - 示例: `adb-apk-installer`, `code-compliance-checker`

- **日志文件名**: `<problem-description>-yyyy-mm-dd.md` (问题描述-年-月-月内序号)
  - 示例: `static-context-leak-2025-02-01.md`, `wrong-patch-label-2025-02-02.md`
  - 说明:
    - `<problem-description>`: 问题描述(非常简洁,使用英文,如: static-context-leak, wrong-patch-label等)
    - `yyyy-mm-dd`: 年-月-日(如: 2025-02-03)

## 日志内容

每个日志文件包含以下章节:

1. **基本信息** - SKILL名称、优化时间、优化人员
2. **问题描述** - 用户反馈、问题分析
3. **相关性判断** - 判断结果、判断依据
4. **问题诊断** - 问题类型、问题位置、问题详情
5. **优化方案** - 优化目标、优化策略、影响评估
6. **优化实施** - 修改的文件、修改详情
7. **验证结果** - 验证状态、验证详情
8. **用户反馈** - 反馈状态、反馈内容
9. **后续计划** - 后续优化计划

## 日志记录原则

1. **每次优化必记录**: 每次执行SKILL优化后,必须创建或更新对应的优化日志文件

2. **月内多次优化**: 由于日志文件名包含月内序号(01, 02, 03, ...),每次优化都会创建新的日志文件,不会出现冲突

3. **完整性**: 必须填写日志文件中的所有章节,确保记录完整

4. **准确性**: 记录的内容必须真实准确,包括修改前后的对比

## 使用场景

- **历史追溯**: 通过日志可以追溯任何SKILL的优化历史
- **问题分析**: 当出现问题时,可以通过日志分析优化过程,找出问题根源
- **模式识别**: 通过分析历史日志,识别常见的优化模式和问题类型
- **策略改进**: 根据日志中的用户反馈,改进优化策略
- **知识积累**: 日志是重要的知识资产,可以作为后续优化的参考

## 特殊说明

- 对于**defect-analyzer**的优化,如果只修改了SKILL.md,日志中不需要特别说明references未修改
- 如果是**自我优化**(skill-optimizer优化自己),SKILL名称为"skill-optimizer"
- 如果优化失败,需要在"验证结果"中详细说明失败原因

## 维护指南

### 添加新日志

当对某个SKILL进行优化时:

1. 检查是否已存在该SKILL的日志目录,如不存在则创建
2. 检查是否已存在当天的日志文件,如存在则在文件末尾追加新的优化记录
3. 如不存在,则创建新的日志文件,填写完整内容
4. 确保日志内容符合格式规范

### 日志管理

- 定期归档历史日志(如超过1年的日志)
- 定期清理无效或错误的日志
- 定期分析日志,提取有价值的信息
- 根据日志分析结果,优化skill-optimizer的策略
